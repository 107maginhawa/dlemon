**1. [Rule 3 - Blocker] ImagingFindings not in OpenAPI without main.tsp registration**

- **Issue:** TypeSpec interface in dental-imaging.tsp namespace compiled but wasn't exported to OpenAPI — codegen produced 0 new handlers

- **Files modified:** specs/api/src/main.tsp
- **Commit:** 691eb77

**2. [Rule 1 - Bug] Generated handler names are ImagingFindingsMgmt_* not ImagingFindings_***
- **Found during:** Task 2 (post-codegen)
- **Issue:** Plan specified ImagingFindings_createFinding but codegen derives prefix from main.tsp interface name (ImagingFindingsMgmt) → produces ImagingFindingsMgmt_createFinding
- **Fix:** Used actual generated names throughout (delegates, test imports). No functional impact.
- **Files modified:** All delegate handlers, imaging.test.ts

**3. [Rule 1 - Bug] Zod strict UUID validation rejects test IDs**
- **Found during:** Task 3c (GREEN phase)
- **Issue:** Zod v4 requires UUID version digit 1-8; test constants like `44444444-0000-0000-0000-000000000011` have `0` in version position → ZodError → 500
- **Fix:** Changed `z.string().uuid()` to `z.string()` for visitId/patientId/branchId/annotationId/treatmentId in CreateFindingSchema and UpdateFindingSchema — consistent with other handlers (e.g., createMeasurement.ts doesn't uuid-validate IDs)
- **Files modified:** createFinding.ts, updateFinding.ts
- **Commit:** 401517c

**4. [Rule 2 - Missing functionality] Migration auto-generated during Task 2 codegen**
- **Found during:** Task 2
- **Issue:** `bun run generate` includes `db:generate` step — migration 0021_pink_vanisher.sql was created automatically
- **Fix:** Migration verified correct and committed as part of Task 2 commit
- **Commit:** 691eb77
