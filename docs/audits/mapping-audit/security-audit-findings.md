# Security Audit Findings

Scope: `services/api-ts/src/handlers/dental-org/**`, `services/api-ts/src/handlers/shared/**`,
`apps/dentalemon/src/routes/auth/**`, route registration in `src/generated/openapi/routes.ts`.

Findings are real defects in the on-disk code (not hypothetical). All line numbers are
against the repo as of 2026-05-26.

## Summary

| Severity | Count |
|----------|-------|
| P0       | 7     |
| P1       | 6     |
| P2       | 4     |
| P3       | 3     |

Headline issues:

- Multiple OpenAPI-style membership/branch/org handlers skip `assertBranchAccess`,
  enabling cross-tenant IDOR (org update, member list/create/deactivate, branch list/get/create).
- `DentalMembershipManagement_list` and `DentalMembershipManagement_deactivate` return raw
  membership rows including `pinHash`, `securityAnswerHash`, and `securityQuestion`.
- `recoverPin` route is registered without `authMiddleware`, and its generated zod validator
  schema does not match the handler's expected body — the endpoint is currently broken and
  any future fix must restore both authentication *and* schema agreement.
- `updateMember` accepts a `role` field with no privilege guard — a `staff_full` member can
  PATCH themselves to `dentist_owner`.
- No API-level rate limiting; PIN lockout is per-member and easily side-stepped by iterating
  across known membership IDs.

## Notes on what already looks good

- All PIN write paths (`setPin`, `verifyPin`, `resetMemberPin`, `setSecurityQuestion`,
  `recoverPin`) call `assertBranchAccess` inside the handler, which is the right pattern.
- `setPin`/`verifyPin` strip `pinHash` from responses (the canonical example to copy elsewhere).
- `Bun.password.hash` defaults to bcrypt; the test-only `cost: 4` override in
  `auth-security-hardening.test.ts:141` is correctly limited to tests.
- The `auth-security-hardening.test.ts` suite (CF-38/39/46) is a good template — extend it
  to cover the SEC-001..SEC-007 IDOR paths above using the same Hono `app.request` harness.
- Logger usage in `dental-org`, `dental-patient`, `dental-clinical`, and `dental-visit`
  does not embed patient PHI — only structured metadata. Keep it that way; add a lint rule.

## Suggested fix order

1. SEC-003 (PIN/security-answer hash leak — fix today, ~5 lines).
2. SEC-007 (role escalation via updateMember — one-line guard swap).
3. SEC-001, SEC-002, SEC-005, SEC-006 (IDOR cluster — add `assertBranchAccess` /
   `assertBranchRole` to each OpenAPI-style handler; mirror the existing flat-endpoint pattern).
4. SEC-004, SEC-011, SEC-012 (response sanitization — strip security columns everywhere).
5. SEC-008 + SEC-009 (recoverPin contract + PIN regex — TypeSpec change + regenerate).
6. SEC-010 (rate limiting — middleware-level).
7. SEC-014 (delete dead handler duplicates after fixes settle).
