## Executive Summary

The dentalemon app has a well-structured monorepo, mature backend test suite (160 real-DB files), and a Playwright journey harness with a hard CI gate. However, **nineteen P0 issues have been identified across all 18 audited modules**, spanning broken core journeys, six CVE-class security vulnerabilities, and critical data-integrity gaps:

1. **Patient navigation** — clicking a patient card uses the wrong route (`/$patientId`), crashing navigation for every user in every role.
2. **Treatment plan tab** — frontend calls `/treatment-plans` (plural) but spec/server expose `/treatment-plan` (singular); tab 404s on every visit.
3. **Recalls tab** — frontend calls `/recalls` endpoints absent from the OpenAPI spec; tab always fails.
4. **Revenue chain (J04)** — invoice creation from the workspace modal always returns 400 because `useCreateInvoice` omits required `branchId` + `dentistMemberId` fields; J04 passes in CI only because of seed state, not real code correctness.
5. **AUTH-01/02/03 (3 security P0s)** — PIN auth issues no server session; `verifyPin`/`setPin` lack cross-org membership verification enabling brute-force; `recoverPin` is a fully unauthenticated endpoint that resets any member's PIN.
6. **Settings F1/F2 (2 security P0s)** — any branch member can deactivate or create members with no owner-only check; privilege escalation is trivially possible.
7. **Perio F1/F2 (2 functional P0s)** — the perio frontend is entirely absent and all 5 perio routes return 404 on the live server.
8. **TypeSpec endpoint auth bypass (2 security P0s)** — `DentalMembershipManagement_create` and `DentalMembershipManagement_deactivate` (the SDK-facing, TypeSpec-generated endpoints) have zero authorization; any authenticated user can create a `dentist_owner` membership in any org or deactivate any membership globally.
9. **Hygienist frontend lockout (1 P0)** — `hygienist` exists in backend `VALID_MEMBER_ROLES` and is used in clinical `assertBranchRole` calls, but is absent from the frontend `ACCESS_MATRIX`; every hygienist member is redirected to `/dashboard` on every navigation — complete UI lockout for a valid clinical role.
10. **Patient branch bypass (1 P0)** — patients with `preferredBranchId = NULL` silently skip `assertBranchAccess` in `getDentalPatient` and `archiveDentalPatient`; any authenticated user can read or archive them. The E2E `patient-checkin.spec.ts` creates patients this way.
11. **Onboarding tenant boundary (2 security P0s)** — `DentalBranchManagement_create` has no org-ownership check (any user can attach branches to any org UUID); `set-pin` endpoint has no ownership check (any user can set any member's PIN).

> **SECURITY ALERT — Do not deploy to any user-facing environment until CF-37–CF-43, CF-81–CF-87 are resolved.** Eleven P0 findings allow unauthenticated PIN reset, cross-org PIN brute-force, privilege escalation, TypeSpec endpoint tenant bypass, and revenue chain blocking.

Beyond P0, the app has a coherent test pyramid but critical gaps: **frontend role gating is entirely absent from the patient workspace** (`role` field from `useOrgContextStore` is read zero times inside workspace), **182 backend unit test files are never run by CI**, **all 35 Hurl contract scenarios are advisory** (`continue-on-error: true`), and **8 of 16 journey specs normalize failures**, creating systematic false confidence.

**18 of 18 modules audited. Overall health: 4.5/10 — backend engine is solid; security hardening, frontend role gating, CI restructuring, and revenue chain fix are all required before deployment.**
