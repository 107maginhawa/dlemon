# All-Module Workflow Traceability Matrix — Dentalemon

**Date:** 2026-06-20
**Lead auditor:** consolidated all-module sweep (COMPLETE — supersedes the earlier 6-module PARTIAL draft)
**Scope:** Every workflow across ALL 11 audited modules — Authentication (better-auth), dental-patient, dental-scheduling, dental-visit, dental-billing, dental-clinical, dental-imaging, dental-pmd, dental-perio, dental-org/audit, and platform/cross-cutting (notifs, comms, storage, email, reviews, emr-consultation, audit-trail, retention, erasure, §12 cross-module flows).
**Method:** Each workflow is graded by what is *actually proven* — distinguishing live e2e persistence proof vs. unit/contract proof vs. shallow/uncovered/not-run/deferred.

---

## 1. Executive Summary

Across the full ~98-workflow surface (129 workflow rows once shipped sub-workflows are counted), the **backend/contract layer is genuinely strong almost everywhere**: the overwhelming majority of workflows are **proven-unit-contract** — real Postgres-backed handler tests through the generated validator chain, plus Hurl contract suites that read durable state back (persistence, FSM transitions, immutability guards, audit-row side effects, role gates) rather than merely asserting a 2xx. By that bar the app is far better-tested than typical. **dental-scheduling is the best-tested module in the repo; dental-billing is the most rigorously tested (~25 backend files with atomic credit/refund rollback, concurrency, FSM, audit read-backs); dental-imaging's ceph path is proven-e2e by four live B-journeys with MinIO up; dental-pmd, dental-org/audit, dental-visit, dental-perio, dental-patient, and dental-clinical all carry deep durable-outcome coverage.**

The **paper-tiger risk is concentrated at the LIVE UI layer.** Only **18 of 129** captured workflows ran live this session and asserted durable goal-state; the rest ride on `.spec.ts` files that exist on disk but are **NOT in the 22-journey harness** — so a broken save button, login form, settings-persist loop, or payment form would pass a 22/22 green run undetected. The two genuine paper-tiger *modules* are **Authentication** (NOT ONE auth workflow runs live, and passkey WF-002 + magic-link WF-003 — the sole patient login path — have zero tests of any kind) and the **platform notification fan-out** (WF-080/082/083/084/085 are [INFERRED] stubs with NO producer wiring; WF-100 EMR-import has no module at all; WF-104 reminder/recall email templates are a pinned Phase-2 gap — armed in-app but not actually emailable).

**The headline doctor-journey finding holds and generalizes at the whole-app level: clinical AUTHORING is the least-proven core.** **WF-011 (SOAP note authoring)** is fully proven at contract+unit (round-trip persistence, locked-immutability) but has **zero journey mapping and no live UI evidence** — no live test ever types a fresh note and reads it back. **WF-009 (dental chart entry)** is backend-proven but its only live journey (J01) asserts *button presence only* and never saves+reads-back a chart entry. So the two acts a dentist performs every single visit — charting a tooth and writing a note — are exactly the two with no live proof the React save path works. Net: **trust the backend logic across all modules (with the FSM-property and auth/notification caveats); do NOT trust that the screens work outside the 18 live-proven workflows** — and note the green dashboard lies loudest because journeys are not a required CI gate, "covered" is computed from map mappings rather than execution, and inverted/DOM-only journeys have historically been counted as health.

## 5. The Honest Tally — Green Can Lie

Three distinct claims must not be conflated:

**(a) "Tests exist."** Almost universally true — every captured workflow except the genuine non-implementations (WF-057 merge stub, WF-061 bulk-slot, WF-P05 print, WF-100 EMR-import) and the auth/notification blind spots (WF-002/003; WF-080/082/083/084/085) has *some* test. This is the layer the green dashboard reflects, and the layer that flatters the app.

**(b) "Tests prove the right thing."** Mostly true at the backend, with sharp exceptions:
- **Mislabeled proof:** WF-001's map points at auth-gates.spec.ts which tests authz gates (BR-016/BR-026), NOT login.
- **Aspirational journey mappings:** "J10 covers WF-038" (J10 drives the visit-NOTE addendum WF-028, not the dental-clinical amendment entity); "J03 covers WF-P02" (J03 stops after the grid renders and never PUTs a perio reading).
- **Property tests that re-declare the map:** billing payment-plan/claim FSM and visit treatment FSM "property" tests validate a transition table copied *into the test file*, not the real repo guard — a guard regression wouldn't flip them red (net risk medium, since separate integration tests cover the guards, but the property test gives false confidence).
- **Shallow job tests / mocked processors:** WF-103 asserts config+exports and never runs the job; WF-104's processor is a pure mock; WF-098's uploadFile.test asserts not-400 not persistence.
- **Contracts that dodge the hard part:** dental-clinical attachment (WF-039) uses a filePath string instead of a real MinIO upload; WF-BIL-REFUND and WF-BIL-CLAIM-FSM have no wire-contract pin at all (integration-only).

**(c) "Tests actually ran (live)."** This is where the dashboard lies loudest. **Only 18 of 129 workflows ran live this session.** Every other UI is proven by a `.spec.ts` that is *on disk but NOT in the 22-journey harness* — patient-registration, patient-profile, data-erasure-admin, patient-checkin, calendar, online-booking, invoice-detail, payment-plan, fee-schedule, prescribe-medication, lab-order-ui, imaging-annotation, imaging-findings, pmd-generation, reporting, add-staff, the auth-pin/auth-gates specs, and more. The map labels many of these "covered." **They produced zero evidence this run.** (The imaging B-journeys and pmd J20 are the bright spots — they ran live with MinIO up and asserted goal-state via independent reads.)

**Specific green-can-lie hazards:**
1. **Journeys are not a required CI gate** (per project memory: the journey gate has been built/armed but not promoted to required branch protection) — even the 18 live journeys can silently stop running without blocking a merge.
2. **Inverted/DOM-only journeys counted as health** is a documented historical failure mode. Any journey asserting only DOM presence (J01 for WF-009) is functionally one of these for the save path.
3. **"Covered" ≠ "ran":** the workflow-test-map marks WFs covered by specs outside the live roster. A reader trusting the map's status column will badly overestimate live UI proof.
4. **API-seeded preconditions** in the live journeys (J22 seeds consent+notes; J04 seeds via API; WF-013 invoice POST is API-driven) mean even the proven-e2e WFs prove the *target action*, not the full authoring chain that precedes it — which is exactly why WF-074 (continuous journey) is the missing keystone.

**Bottom line:** Trust the backend logic across all 11 modules (with the FSM-property, auth, and notification-fan-out caveats). Do NOT trust that the screens work for anything outside the 18 live-proven workflows — concentrated in scheduling (J17), visit (J04/J12/J21/J22), billing (J04), imaging (B01-B04), pmd (J20), perio (J03), and org/audit (J10/J18). The single highest-leverage fix is one continuous live doctor-journey (gap #4) that forces WF-009 chart save and WF-011 note save through the real UI — closing the two least-proven core acts and the keystone gap at once — and promoting the journey suite to a required gate so the honest live proofs can't silently rot.
