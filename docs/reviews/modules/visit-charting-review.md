# Visit Workspace + Dental Charting — Standards & Experience Review
> Review date 2026-06-02 · Depth: DEEP · Carousel centerpiece

## 1. What we have   (cite spec + code paths)

The visit workspace is the core clinical surface (`docs/product/modules/dental-visit/MODULE_SPEC.md`). It pairs a **Timeline Carousel** of per-visit dental charts with a clinical treatment table (flex-row, ADR-005). Verified against code:

- **Timeline Carousel** — Swiper `EffectCoverflow` Cover Flow browser sorted oldest→newest, newest as `initialSlide`; each slide is a self-fetching `VisitChartCard` (`apps/dentalemon/src/features/workspace/components/timeline-carousel.tsx`). Active card is fully interactive (`toothSize='md'` ≈ 80px per `dental/types.ts`), neighbors are preview-only (`xs` ≈ 28px). Keyboard module enabled; pagination dots clickable.
- **Dental Chart** — SVG anatomical crown+root teeth (not box odontogram) rendered as native `<button>` per tooth, 4 quadrants, FDI layout (`dental-chart.tsx`, `dental/universal-tooth.tsx`, `public/teeth/*.svg`). Color-by-state via `getToothFillColor` (Apple-system palette). A **segmented layer control** (Baseline / Proposed / Completed) shows one layer at a time; off-layer teeth dim to 0.2 opacity; proposed teeth get a dashed outline (CR-03, CHART-BR-006).
- **Tooth Slideout** — 3-step wizard Overview → Treatment → Review (`tooth-slideout.tsx`, `tooth-overview-step.tsx`). Per-surface condition assignment via a B/D/M/P/O ring + pills (`five-surface-selector.tsx`), 9 condition chips with color dots (Caries/Fracture/Crown/Extract/Filling/Missing/Implant/Watchlist/Healthy), an **Entry Classification** picker (Existing / Existing-Other "From another provider" / Treatment Plan / Condition), and a per-tooth Treatment Breakdown history table (screenshot `docs/reviews/screenshots/04-tooth-slideout.png`).
- **Save flow** — `use-save-tooth-flow.ts` saves chart first, then treatment (CR-04 ordering so a treatment is never silently dropped). `use-initialize-dentition.ts` → `POST /dental/patients/:id/dentition` seeds the chart by age.
- **Backend** — `services/api-ts/src/handlers/dental-visit/`: visit FSM (BR-001/002/003), treatment FSM forward-only incl. `declined`/`dismissed` (`repos/treatment.schema.ts` `TREATMENT_TRANSITIONS`), chart upsert (`chart/upsertDentalChart.ts`), and a **per-patient cumulative baseline** `dental_patient_chart_baseline` that merges each visit's chart last-write-wins with baseline-immutability (CHART-BR-002, `repos/dental-chart-baseline.repo.ts`, `dental-chart-baseline.test.ts`).

## 2. Industry-standard benchmark

Leading PMS (Open Dental, Dentrix, Curve, CareStack, tab32, Eaglesoft) converge on: a single Chart module combining graphical odontogram + treatment entry + progress notes; **configurable tooth notation** (Universal/FDI/Palmer/Haderup) with stable tooth identity; **five-surface** multi-surface charting; **independently filterable status layers** (treatment-planned, completed, existing-own, existing-other, referred, condition) with blue=existing/done / red=planned color convention; **mixed dentition shown simultaneously**; a **date slider for chart time-travel** (a time-indexed projection of dated procedure events); per-procedure clinical notes; audit-trail + note signing + provider attribution; and the rising **voice/AI charting** frontier. A dedicated odontogram before/after *carousel* is a differentiator, not table-stakes. Full sourced detail: `../research/visit-charting.md`.

## 4. UX/UI assessment   (Nielsen + WCAG 2.2 AA + DESIGN.md)

**Strengths.** The slideout (`04-tooth-slideout.png`) is genuinely best-in-class for the entry path: clear 3-step progress indicator, anatomical surface ring with B/D/M/P/O labels, color-dotted condition chips, and an Entry Classification that closes the common "other-provider attribution" gap most products bury. Visibility of system status is strong (loading skeletons, error+retry on each card, lock confirmation, sync badge). DESIGN.md fidelity is high — #FFE97D lemon accent on the active card border/accent-bar and the active layer tab, Apple-system condition colors, rounded-2xl cards, soft shadows. Touch targets meet the 44px iPad rule (ADR-001) on chrome buttons; condition/surface pills are generous.

**Heuristic issues.**
- *Recognition vs recall / Match to real world (Nielsen #2,#6):* the chart hard-renders FDI even when a US branch sets Universal notation in Settings — the displayed numbers won't match the clinician's mental model. The notation control is a dead-end preference (§3 P1).
- *Flexibility (Nielsen #7):* layers are mutually exclusive tabs; clinicians cannot see existing + planned together, which is the normal "what's there vs what I propose" comparison. Mixed dentition can't be viewed as one arch.
- *User control & freedom (Nielsen #3):* **Escape does not dismiss the slideout or the Rx/lab sheets** (live-audit had to reload). No confirm on destructive condition charting (extract/missing).
- *Error prevention (Nielsen #5):* no left/right orientation cue on the chart — a documented wrong-tooth/adjacent-tooth error class.

**WCAG 2.2 AA.**
- **4.1.2 Name/Role/Value — reconcile:** each tooth in `dental-chart.tsx` IS a native `<button>` with a rich `aria-label` ("Tooth 18: Upper Right Third Molar, watchlist, baseline") and `aria-pressed`, so it carries the *implicit* button role and will announce as a button to screen readers. The live-audit's "0 `role="button"` elements on page" is an **attribute-presence** observation (no *explicit* `role="button"`), not a true role gap — native buttons don't need it. **No P0 here.** The genuine risk is the inner `UniversalTooth` carrying `role="img"` + an `aria-label` *inside* the button, producing a nested/duplicated announcement; that is a P2 polish item, not the operability blocker the raw screenshot count implied.
- **2.1.1 Keyboard / 2.1.2 No keyboard trap:** carousel has Swiper Keyboard enabled (good); but slideout/sheets lack Escape-to-close and there is no visible focus-management/return-focus on open/close (P2).
- **1.4.3 Contrast:** layer-toggle active text `#4A4018` on `#FFE97D` is fine; dimmed off-layer teeth at 0.2 opacity may fail non-text contrast if used as the only differentiator (P3).

## 5. Findings (P0–P3)

- **[P1] Notation toggle is wired in Settings but ignored by the chart** — `locale-settings.tsx` persists `toothNotation` (FDI/Universal/Palmer) per branch, yet `dental-chart.tsx` always renders `UniversalToothFdi` with FDI numerals and never reads the setting. — *Recommendation:* thread `toothNotation` (branch settings) into `DentalChart`/`UniversalTooth` `label` so US branches see Universal numbering; keep FDI as the stable internal identity. This is also flag `dental_visit_universal_notation` (MODULE_SPEC §18) currently inert.
- **[P1] Status layers are mutually exclusive, not filterable overlays** — Baseline/Proposed/Completed is a one-at-a-time segmented control (`dental-chart.tsx:126-151`), unlike the industry "toggle any combination" Show-tab. — *Recommendation:* allow multi-select layer visibility (existing + planned together) and add an own-vs-other-provider distinction on the baseline layer.
- **[P1] Mixed dentition not rendered** — backend seeds true mixed dentition (deciduous + permanent, age 6–12, `initializeDentition.ts:80-85`) but the frontend `getDentitionType` is binary and `DentalChart` renders only primary OR permanent — the mixed data is dropped on the floor. — *Recommendation:* add a `mixed` render path (overlay/interleave primary+permanent per quadrant) + a primary/mixed/permanent view switch.
- **[P2] No whole-chart before/after comparison** — the carousel sequences visit cards but cannot diff two snapshots side-by-side; per-tooth history is a flat table. The cumulative `dental_patient_chart_baseline` exists and could power this. — *Recommendation:* a 2-up snapshot compare (and per-tooth delta) leveraging the baseline + per-visit charts.
- **[P2] No in-card date slider / explicit time-travel** — time-travel is implicit in carousel navigation only. — *Recommendation:* optional date scrubber that projects the cumulative baseline as of a chosen date.
- **[P2] Escape does not close slideout / Rx / lab sheets; no destructive-action confirm; no L/R cue** — Nielsen #3/#5 + WCAG 2.1.2 hygiene. — *Recommendation:* add Escape handler + focus return on the `<aside>` panels; confirm dialog for extract/missing charting; render a small L/R orientation marker on the arches.
- **[P2] Own-vs-other-provider not visually distinguished on the chart** — captured in Entry Classification but not surfaced as a chart layer/badge. — *Recommendation:* badge/legend "Existing (other provider)" on the baseline layer.
- **[P3] Color convention diverges from blue=done / red=planned norm; no RCT/endo symbol; per-procedure note threading absent; no voice charting** — all acceptable for MVP; note for parity roadmap.
- **[P3] Nested SR announcement** — inner `UniversalTooth` `role="img"`+aria-label inside the button tooth may double-announce. — *Recommendation:* set the inner SVG wrapper `aria-hidden` since the button already names the tooth.

## 6. Carousel implications   (how this shapes the carousel + per-tooth timeline)

- **The signature story is undemonstrated.** The demo seed has **one visit per patient**, so the Cover Flow shows a single card with no neighbors (`03-workspace-carousel.png`) — the multi-visit before/after comparison that is the product's central differentiator is invisible. *This is the single highest-leverage fix:* seed each demo patient with 3–5 chronological visits showing chart evolution (caries → planned → performed across dates). Without it, eng-strong infrastructure (per-card lazy chart fetch, cumulative baseline, layer model) reads as a static chart.
- **The data model already supports the carousel's promise.** Per-visit charts + the cumulative `dental_patient_chart_baseline` (last-write-wins merge with baseline immutability) are exactly the time-indexed projection the research validates. The gap is *presentation*: there's no diff/compare view and no date scrubber to make the longitudinal model legible.
- **Layer model is the carousel's per-card lens.** Baseline/Proposed/Completed within the active card is a strong concept, but mutually-exclusive tabs blunt the "existing vs proposed" comparison the carousel is meant to dramatize. Making layers multi-select would let a single card tell the before→after story even when only one visit exists.
- **Per-tooth timeline is half-built.** The slideout's Treatment Breakdown table is the per-tooth longitudinal record, but it's tabular, not a visual timeline, and doesn't tie back to carousel positions. A per-tooth mini-timeline (visit dots → state changes) would unify the whole-chart carousel with the per-tooth view.
- **Notation + mixed-dentition gaps cap international/pediatric demos.** A US or pediatric patient can't be shown correctly today (FDI-only render, no simultaneous mixed view), so the carousel's "any patient, any locale" framing is currently limited to adult FDI cases.
