## Problem Statement

No dental-native practice management system serves the global market affordably. Dentrix (US-only, subscription-locked) dominates North America. Emerging market alternatives (MYCURE, Mediks, Molarsoft) are general clinic systems adapted for dental — they lack native charting, the #1 purchase driver. Dentists compensate with workarounds: extra staff, parallel paper records, and memorizing charts because their software has no visual chart navigation.

Dentalemon v2 PRD (1521 lines) captured the product vision comprehensively but embedded tech stack references throughout (Tauri, Swiper.js, Cadence/Iroh/Loro, SQLite, React, shadcn/ui) and missed several clinical workflows, edge cases, and global compliance requirements. The v3 rewrite makes the PRD purely product-focused, closes identified gaps, and adds PMD (Portable Medical Document) compliance for health record interoperability.

## Key Definitions

**Timeline Carousel:** A horizontal carousel of visit cards where each card shows a dental chart preview (visual "fingerprint" of tooth states). The UX selling point — browsing visits like album covers.

**Per-Tooth Timeline:** A chronological history view for a single tooth across all visits — every condition, treatment, and image tagged to that tooth. The data moat — no competitor does this well.

**Smart Attachments:** Images (x-rays, intraoral photos) uploaded to a visit with dental-aware tagging: image type (periapical, bitewing, panoramic, intraoral photo) + tooth number(s) the image relates to. Viewable in a lightbox with zoom/pan. When viewing a tooth, shows "this tooth has N images across M visits."

**Safety Floor:** A PMD concept — an always-visible panel showing a patient's critical safety information: known allergies, current medications, and active medical conditions. Data comes from the app's own records + any imported PMDs from other providers.

**PMD (Portable Medical Document):** An open document format for portable health records. One completed visit = one PMD file. Digitally signed, works offline, patient-carried. Enables record exchange between any PMD-compatible providers without vendor lock-in or network dependency.

**Lab Order:** Manual tracking of prosthetic work sent to dental labs: lab name, order date, expected delivery date, status (ordered/in-fabrication/delivered/fitted). Phase 1 is tracking only — no automated communication with labs. State machine: Ordered → In Fabrication → Delivered → Fitted. Can also go Ordered → Cancelled (with reason). Cannot skip states. Cannot go backward (if lab returns defective work, create a new order linked to original).

## Edge Case Business Rules

| # | Edge Case | Resolution |
|---|-----------|------------|
| EC1 | Archived patient with active payment plan | Block archive. Show "Patient has ₱X outstanding. Settle or write off before archiving." |
| EC2 | Pending treatments on extracted tooth | Auto-dismiss with reason "Tooth extracted." Logged in audit trail. |
| EC3 | PWD + Senior double discount (PH) | Only highest applies. Rule: "One discount per line item, highest eligible." |
| EC4 | Treatment fee changes mid-visit | Price at time of treatment recording is locked. Fee schedule changes apply prospectively only. |
| EC5 | Voided payment receipt reprint | "VOIDED" watermark on reprint. Original receipt preserved in payment history. |
| EC6 | Multiple payment plans per patient | Allowed. Each plan independent. Dashboard shows all active plans with aggregate status. |
| EC7 | Maximum open visits per patient | 1 active visit per patient per device. Second device gets "Visit already active on [device name]." Note: two devices CAN edit the same patient record (e.g., assistant updates address while dentist has workspace open) — the restriction is on creating a NEW visit, not on concurrent access to the patient. |
| EC8 | Currency rounding on discounts | Round to nearest centavo (2 decimal places). Banker's rounding (round half to even). |
| EC9 | Patient turns 16 during treatment | System prompts "Patient is now 16. Update consent to self-consent?" at next visit check-in. |
| EC10 | Backup restore vs newer local data | Sync engine merges automatically. If no sync: restore warns "Local data is newer. Overwrite?" |
| EC11 | Invoice number gaps from device prefix | Accepted. Per-device prefix ensures uniqueness. Gaps are expected in distributed systems. |

## Conflict Resolution Model

Multi-device conflict resolution at the product level:
- **Field-level merge:** Concurrent edits to different fields on the same record merge automatically (e.g., assistant updates address while dentist records treatment)
- **Same-field conflict:** Last-write-wins with full audit trail (both values preserved in history)
- **Clinical safety rule — treatments:** Treatment records are append-only. Two devices recording treatments on the same tooth = both records kept, flagged for clinician review
- **Clinical safety rule — Safety Floor:** Allergies, medications, and active conditions use add-only merge. A deletion on one device does NOT remove an allergy added on another device. Removals require explicit clinician confirmation after sync. This prevents silent loss of safety-critical data.
- **User visibility:** Sync status indicator in app shell (synced / syncing / offline / conflict). Conflicts never block clinical workflow.
- **PMD immutability boundary:** PMD is generated when visit status transitions to "Completed." After PMD generation, the visit's clinical data is locked (amendments only). CRDT merges that arrive after PMD generation create amendment records, not overwrites. A new PMD can be generated that supersedes the previous one (linked via provenance).

## Reviewer Concerns

Issues identified by adversarial review that were not fully resolved (2 rounds, 16 issues found, 12 fixed):

1. **CDT mapping count:** 50 most common procedures may be undersized for a dentist doing restorative + endo + perio + prosth + oral surgery (100-150 codes routinely used). Consider shipping with 100+ mapped codes or providing an easy mapping interface.
2. **PMD error states:** Corrupted PMD files, failed signature verification, version mismatch with newer PMD specs — these need error handling specified in the PRD's PMD section.
3. **Offline digital signatures:** Certificate validation typically requires network. The PRD should specify how signatures work offline (pre-loaded facility certificates, local validation).
4. **Smart Attachment storage limits:** Image compression strategy, maximum file sizes, and supported formats should be specified in the PRD alongside the Smart Attachments feature.
5. **"Tech-agnostic" boundary definition:** The PRD says "zero framework references" but uses terms like "CRDT," "Apple Pencil," "Split View," "3D transforms." Rule: platform capabilities (Apple Pencil, Split View) are product requirements. Algorithmic concepts (CRDT) describe behavior, not implementation. Rendering techniques (3D transforms) should be described as visual effects, not tech.
