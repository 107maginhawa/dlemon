# Module Specification: dental-scheduling

---
Spec Version: 1.0 | Last Updated: 2026-05-24
---

## 1. Module Overview
**Purpose:** Appointment lifecycle management. Supports create/reschedule/check-in/cancel with soft double-booking at create (warn) vs. hard block at reschedule (409). Check-in creates a linked Visit record that outlives the appointment.

**Users:** staff_full, staff_scheduling (scheduling ops); dentist_owner, dentist_associate (check-in, view); staff_scheduling (calendar view only)

**Related:** dental-org (working hours, assertBranchAccess), dental-visit (visit creation on check-in), dental-patient (patient lookup), notifs (appointment notifications), booking (slot generation — G-001)

---

## 2. Domain Terms

| Term | Definition |
|------|-----------|
| Appointment | Scheduled encounter slot for a patient+dentist at a branch |
| Check-in | Converting a scheduled appointment to an active Visit (BR-004) |
| Walk-in | Appointment created same-day bypassing slot availability (BR-SCH-002) |
| Double-booking | Two appointments for same dentist in same time window |
| Slot | Available booking window (generated by booking module pg-boss job) |

## 4. Workflow Details
### WF-007: Check-in
Actor: staff_full | Steps: 1. Staff selects appointment 2. Click "Check-in" 3. System verifies BR-001 (no active visit for patient) 4. Creates Visit draft → active 5. Returns visitId | Exception: BR-001 violated → 409, prompt staff

### WF-006: Book Appointment
Actor: staff_full | Steps: 1. Select date/time/dentist 2. System checks for overlap 3. If overlap: warn DOUBLE_BOOKING but proceed (FR3.7) 4. Save | Reschedule: hard 409 on overlap

## 5. Business Rules

| Rule ID | Rule | Expected Behavior |
|---------|------|-------------------|
| BR-004 | Check-in creates visit; appointment delete ≠ visit delete | Visit persists after appointment cancel |
| BR-SCH-001 | All appointments scoped to branch | assertBranchAccess |
| BR-SCH-002 | Walk-in bypass slot availability | Skip working-hours check if walk_in=true |
| BR-SCH-003 | Cancellation requires reason | cancellation_reason NOT NULL |
| BR-SCH-004 | Appointments validate against branch working hours | 422 if outside hours (unless walk-in) |
| FR3.7 | Double-booking: soft-warn at create, hard-block at reschedule | Create → 201+warning; Reschedule → 409 |

## 7. Data Requirements
**`dental_appointment`:** id, patient_id, branch_id, dentist_member_id, scheduled_at, duration_minutes, status (enum: scheduled/checked_in/completed/cancelled/no_show), walk_in (bool), cancellation_reason, cancelled_at, checked_in_at, visit_id (nullable, set on check-in)

---

## 7b. Aggregate Boundaries
Appointment is an aggregate root. Visit referenced by visit_id (UUID only) after check-in — Visit is owned by dental-visit.

---

## 8. State Transitions
```
scheduled ──► checked_in ──► completed
scheduled ──► cancelled
scheduled ──► no_show
```
No DB unique constraint on dentist+time (see FR3.7 — intentional).

## 11. Acceptance Criteria
**AC-SCH-001:** Double-booking at create → 201 + DOUBLE_BOOKING warning in response body.
**AC-SCH-002:** Double-booking at reschedule → 409.
**AC-SCH-003:** Check-in with existing active visit for patient → 409.
**AC-SCH-004:** Cancel without reason → 422.
**AC-SCH-005:** Appointment cancelled → associated visit still accessible.

---

## 12. Test Expectations
Unit: FR3.7 (soft-warn vs hard-block), BR-SCH-004 working hours, BR-004 visit survival.
Integration: check-in creates visit in dental-visit module.

---

## 13. Edge Cases
- No-show appointment → status no_show; no visit created
- Walk-in outside working hours → allowed (BR-SCH-002)
- Appointment for patient at wrong branch → 403
- Slot generation job not implemented (G-001) → manual appointment creation only

---
