## Findings considered and rejected (so nobody re-audits them)

- **All 2026-06-16 audit security P0/P1s** (P0-1/2, P1-1..7): verified FIXED in
  current source, each with a comment citing the audit's own numbering. Not
  re-planned.
- **P1-8 PII in interpolated logs**: FIXED — `core/auth.ts` logs email as a
  redacted object field, not an interpolated message string.
- **P1-9 RLS activation on ~60 unarmed tables**: by-design deferral, documented

  activation at multi-tenant cloud launch). Settled decision, not a finding.
- **H-2 Stripe webhook 200-on-failure / idempotency**: FIXED — webhook now
  returns retryable 5xx on processing failure and has an idempotency ledger with
  a replay test (`handleStripeWebhook.test.ts`).
- **Payments/erasure/legal-hold "orphan endpoints"**: NOW COVERED — 8 dedicated
  test files (`payInvoice`, `captureInvoicePayment`, `refundInvoicePayment`,
  `handleStripeWebhook`, `placeLegalHold`, `approveErasure`, `mergePatients`).
- **P1-7 storage stored-XSS download vector**: the live vector is CLOSED
  (`core/storage.ts` forces `Content-Disposition: attachment` +
  `application/octet-stream`). Residual = no upload-side mimeType allowlist =
  defense-in-depth only; low value, not planned (revisit if downloads ever serve
  `inline` again).
- **`expand.ts` N+1**: acknowledged design tradeoff (per-item authorization);
  not worth changing without a batch-auth endpoint. Low value.
- **`alert()` in `personal-info-form.tsx`, `joinVideoCall` `USE_SESSION_TOKEN`
  sentinel, FE invoice-invalidation missing `branchId` (auto-corrected by
